Структурні патерни відповідають на запитання **"як з'єднувати об'єкти"** — як скласти з простих елементів складніші структури, як адаптувати несумісне, як приховати складність.

## Перелік патернів

| Патерн | Завдання у ЛР3 | Що розв'язує |
|---|---|---|
| Adapter         | Уніфікація платіжних API     | Несумісні інтерфейси |
| Facade          | Оформлення замовлення        | Спрощення складної підсистеми |
| Proxy           | Lazy-завантаження фільмів    | Заступник з контролем доступу |
| Composite       | CRM-модулі                   | Робота з листям і деревом однаково |
| Decorator       | SaaS-документи               | Додавання поведінки без зміни класу |
| Bridge          | Аналітика × Відображення     | Розділення абстракції від реалізації |
| Flyweight       | Minecraft-блоки              | Економія пам'яті через спільний стан |

## 1. Adapter — Платіжні провайдери

**Завдання**: інтегрувати JSON, XML та async-crypto провайдерів під єдиний інтерфейс `IPaymentProvider`.

**Структура**:
```
IPaymentProvider                    (target — те, що очікує ядро)
   ↑
JsonPaymentAdapter ── обгортка ──→ JsonPaymentService (adaptee)
XmlPaymentAdapter  ── обгортка ──→ LegacyXmlPaymentProvider
CryptoPaymentAdapter ── обгортка ──→ AsyncCryptoProvider
```

**Що цікаво**:
- `CryptoPaymentAdapter` синхронно "приховує" async-API через `.GetAwaiter().GetResult()`. Adapter не лише адаптує дані — він може адаптувати модель виконання.
- Кожен adapter — окремий клас, ядро не знає про їх існування.

**Принципи**: SRP (один адаптер — одна інтеграція), OCP (новий провайдер = новий клас адаптера).

## 2. Facade — Оформлення замовлення

**Завдання**: 6 підсистем (Pricing, Discount, Inventory, Payment, Shipping, Notification) — клієнт повинен оформляти замовлення одним викликом.

**Що приховує фасад**:
- Порядок виклику (Inventory check → Pricing → Discount → Reserve → Charge → Schedule → Notify).
- Залежності між підсистемами.
- Обробку помилок (склад порожній → виняток).

**Тонкий момент**: фасад НЕ забороняє доступ до підсистем напряму. Якщо клієнту потрібен лише `PricingService` — він може звернутися безпосередньо. Facade — це **зручність**, а не **обмеження**.

**Принципи**: Law of Demeter (клієнт говорить лише з фасадом), SRP (фасад відповідає за орхестрацію).

## 3. Proxy — Lazy завантаження фільмів

**Завдання**: каталог фільмів. Більшість користувачів бачить лише назви. Деталі — важкі. Преміум-контент потребує перевірки.

**Три ролі Proxy в одному класі**:

| Тип | Реалізація у `MovieProxy` |
|---|---|
| Virtual Proxy   | `Load()` створює `Movie` лише при першому зверненні до важких даних |
| Cache Proxy     | `_cached` зберігає результат |
| Protection Proxy | Перевірка `_userHasPremium` перед `Load()` |

**Чому це нормально їх об'єднувати**: усі три варіанти Proxy — це варіанти ОДНОЇ ідеї "посередник, що зберігає інтерфейс об'єкта".

**Принципи**: SRP (Proxy має ТРИ обов'язки — це межа припустимого), OCP (новий тип Proxy додається без зміни Real Subject).

## 4. Composite — CRM-модулі

**Завдання**: модулі CRM можуть утворювати дерева ("Інтеграції" → "Slack", "Telegram", "1С").

**Ключова ідея**: листя (`LeafModule`) і композити (`CompositeModule`) мають **спільний інтерфейс** (`CrmModule`). Клієнт викликає `crm.GetTotalPrice()` і не знає, чи `crm` — окремий модуль чи ціла гілка.

**Каскадне вимкнення**:
```csharp
public override void Disable() {
    base.Disable();
    foreach (var c in _children) c.Disable();        // ← рекурсія
}
```

**Принципи**: OCP (новий тип модуля = новий підклас Component), LSP (Leaf і Composite взаємозамінні з точки зору клієнта).

## 5. Decorator — SaaS-документи

**Завдання**: документ комбінується з функцій (шифрування, стиснення, водяний знак, аудит).

**Чому НЕ успадкування**:
```
PlainDocument
EncryptedDocument
CompressedDocument
EncryptedCompressedDocument
WatermarkedEncryptedDocument
WatermarkedCompressedDocument
...
```
4 функції → 2⁴ = 16 класів. 5 функцій → 32.

**Decorator замість успадкування**: 1 клас на функцію, динамічне склеювання через композицію:

```csharp
new AuditLogDecorator(
    new EncryptionDecorator(
        new CompressionDecorator(
            new WatermarkDecorator(
                new PlainDocument("contract.docx"),
                "КОНФІДЕНЦІЙНО")),
        key: "AES-256"),
    user: "stepanenko@kpi.ua");
```

**Тонкий момент**: порядок декорування важливий — шифровані дані не стискаються. Це не недолік Decorator — це його сила (порядок визначає клієнт).

**Принципи**: OCP (нова функція = новий декоратор), SRP (один декоратор — одна функція), композиція над успадкуванням.

## 6. Bridge — Аналітика × Відображення

**Завдання**: 3 типи даних × 3 рендерера = 9 комбінацій. Класичний combinatorial explosion.

**Розв'язання**: дві незалежні ієрархії, з'єднані композицією:

```
   AnalyticsView (Abstraction)         IRenderer (Implementor)
   ├ FinancialMetricsView              ├ WebDashboardRenderer
   ├ UserMetricsView         ←──→      ├ MobileAppRenderer
   └ LogVolumeView                     └ PdfExportRenderer
```

Тепер новий тип даних = +1 клас. Новий рендерер = +1 клас. Не 9.

**Bridge vs Adapter**: Adapter з'єднує **існуючі** несумісні класи, Bridge **проєктується** з самого початку для розділення.

**Принципи**: OCP, DIP (Abstraction залежить від Implementor через інтерфейс).

## 7. Flyweight — Minecraft-блоки

**Завдання**: 100 000+ блоків у світі. Кожен має текстуру/модель. Копіювати їх для кожного блока — overload пам'яті.

**Розділення стану**:

| Стан | Тип | Розташування | Приклад |
|---|---|---|---|
| Intrinsic (спільний) | `BlockType` | У `BlockTypeFactory.Pool` | Текстура, модель, базова міцність |
| Extrinsic (унікальний) | `WorldBlock` | На кожному екземплярі | Позиція, поточна міцність |

**Фабрика-кеш** — критична. Без неї Flyweight не Flyweight:

```csharp
public static BlockType Get(string name) {
    if (!Pool.TryGetValue(name, out var bt))
        Pool[name] = bt = new BlockType(name, ...);
    return bt;                                       // повертаємо ТЕ САМЕ посилання
}
```

У демо: 100 000 блоків, тільки **5** реальних BlockType.

**Принципи**: SRP (BlockType — лише спільні дані, WorldBlock — лише унікальні).

## Загальні принципи структурних патернів

1. **Композиція > успадкування** — більшість патернів цієї категорії передають поведінку через композицію об'єктів.
2. **Інтерфейси, не реалізації** — клієнти залежать від абстракцій (DIP).
3. **Розділення обов'язків** — кожен патерн відокремлює якийсь аспект.

## Шпаргалка

| Сценарій | Патерн |
|---|---|
| Несумісні API → єдиний контракт | Adapter |
| Багато підсистем → одна точка входу | Facade |
| Замінник для контролю/lazy/cache | Proxy |
| Дерево однотипних елементів | Composite |
| Динамічна комбінація функцій | Decorator |
| Два виміри змінюваності | Bridge |
| Мільйони об'єктів зі спільним станом | Flyweight |
