# Породжувальні патерни (Creational Patterns)

> Лабораторна №2. Курс ООА.
> Код: [University-Repository/.../OOA/LR/LR2](../../../University-Repository/Year-2/Semester-2/OOA/LR/LR2)
> Дедлайн: 17.03.2026 ✅

Породжувальні патерни відповідають на запитання **"як створювати об'єкти"** — а саме як уникнути жорсткої прив'язки до конкретних класів, як побудувати складний об'єкт із кроків, як гарантувати єдиність екземпляра.

## Перелік патернів

| Патерн | Завдання у ЛР2 | Що розв'язує |
|---|---|---|
| Builder            | Створення персонажа гри          | Складна покрокова конструкція |
| Abstract Factory   | UI-теми (темна/світла)            | Сімейство пов'язаних продуктів |
| Singleton          | Сервіс логування                  | Єдиний екземпляр на додаток |
| Prototype          | Орки у грі-стратегії              | Швидке клонування з шаблона |
| Factory Method     | Експорт звітів (PDF/Excel/HTML/JSON) | Динамічний вибір типу продукту |

## 1. Builder — Творець персонажа

**Завдання**: гра дозволяє створити персонажа з ім'ям, статтю, класом, статами, кольором очей, зачіскою, зброєю. Потрібно і "стандартний боєць", і повна кастомізація.

**Чому Builder**:
- Багато параметрів (9 полів) — конструктор перетвориться на телефонний номер.
- Деякі поля опціональні з розумними дефолтами.
- Потрібні "пресети" (`createDefaultWarrior()`) і кастомні комбінації.

**Структура**:
```csharp
Character             // Product (immutable, init-only)
CharacterBuilder      // Builder з fluent API (.SetName().SetClass()...)
CharacterDirector     // Знає "рецепти" типових персонажів
```

**Ключовий момент**: усі сетери повертають `this` → ланцюжкові виклики `.SetX().SetY().Build()`.

**Зв'язок із SOLID**: SRP (Director знає рецепти, Builder — як збирати, Character — лише дані), OCP (новий тип персонажа = новий метод у Director).

## 2. Abstract Factory — UI-теми

**Завдання**: усі елементи інтерфейсу (кнопка, текстове поле, список) повинні бути або всі темні, або всі світлі. Не можна змішувати.

**Чому Abstract Factory**:
- Створюємо **сімейство** об'єктів — не один, а кілька пов'язаних.
- Гарантуємо узгодженість: одна фабрика — одна тема.
- Клієнт працює з інтерфейсами (`IButton`, `ITextField`) і не знає, темна вони чи світла.

**Структура**:
```csharp
IUIFactory                         // Abstract Factory
   ├ DarkThemeFactory              // Concrete Factory
   └ LightThemeFactory             // Concrete Factory

IButton ── DarkButton / LightButton
ITextField ── DarkTextField / LightTextField
IDropdown ── DarkDropdown / LightDropdown
```

**Відмінність від Factory Method**: Factory Method створює ОДИН тип продукту, Abstract Factory — СІМЕЙСТВО.

**Зв'язок із SOLID**: DIP (Application залежить від `IUIFactory`, не від конкретних класів), OCP (нова тема = нова фабрика).

## 3. Singleton — Сервіс логування

**Завдання**: усі модулі мають писати логи в одне місце. Множинні підключення = проблема.

**Чому Singleton**:
- Глобальна точка доступу.
- Гарантовано один екземпляр (одне підключення до файла/сервера).

**Реалізація з потокобезпекою**:
```csharp
public sealed class LogService {
    private static LogService? _instance;
    private static readonly object _lock = new();
    private LogService() {}                          // приватний конструктор
    public static LogService Instance {
        get {
            if (_instance is null) {
                lock (_lock) { _instance ??= new(); }
            }
            return _instance;
        }
    }
}
```

**Перевірка єдиності**: `ReferenceEquals(authLogger, dbLogger)` → `true`.

**Контроверсії Singleton**:
- Глобальний стан — складніше тестувати.
- Прихована залежність (метод бере `LogService.Instance` усередині, а не з конструктора).
- Альтернатива у сучасному коді — DI-контейнер із `Singleton` lifecycle.

**Зв'язок із SOLID**: DIP порушено навмисне (Singleton сам по собі — конкретний клас). У продакшені краще використовувати DI з singleton-реєстрацією.

## 4. Prototype — Клонування орків

**Завдання**: на карті 1000 орків. Кожен має ту саму текстуру/модель, але різну позицію/зброю/рівень. Створювати з нуля — повільно.

**Чому Prototype**:
- Уникаємо важкого конструктора (завантаження ресурсів) при кожному створенні.
- Маємо "еталонного" орка у пам'яті — клонуємо.
- Спільні дані (текстура, ефекти) діляться між клонами через посилання (це fly­weight-ефект).

**Ключова деталь**: `Clone()` копіює value types та посилання на ресурси (intrinsic), а змінні поля (позиція, рівень) клієнт встановлює окремо.

**Зв'язок із Flyweight**: Prototype + Flyweight часто йдуть разом. У ЛР3 Flyweight реалізовано окремо для Minecraft-блоків.

## 5. Factory Method — Експорт звітів

**Завдання**: програма генерує звіти у різних форматах (PDF, Excel, HTML, JSON). Тип обирається з налаштувань.

**Чому Factory Method**:
- Клієнт не повинен робити `new PdfReport()` — лише питати "дай мені експортер".
- Логіка вибору типу інкапсульована у фабриці.

**Структура**:
```csharp
abstract class ReportFactory {
    public abstract IReport CreateReport();          // Factory Method
    public void ExportReport(string data) {           // Template Method
        var r = CreateReport();
        r.Generate(data);
    }
}
class PdfReportFactory : ReportFactory { ... }
class ReportFactoryResolver { static GetFactory(format) => switch... }
```

**Цікаво**: у моїй реалізації Factory Method поєднано з Template Method (`ExportReport()` — алгоритм, `CreateReport()` — крок, що змінюється). Це часта комбінація.

**Зв'язок із SOLID**: OCP (новий формат = новий клас фабрики), DIP (клієнт залежить від `ReportFactory`).

## Шпаргалка: коли який патерн?

| Ситуація | Патерн |
|---|---|
| Багато опціональних параметрів конструктора | Builder |
| Створюю сімейство сумісних об'єктів | Abstract Factory |
| Потрібен один екземпляр на додаток | Singleton (з обережністю) |
| Об'єкт важко створити, але легко клонувати | Prototype |
| Вибір типу продукту з конфігурації / `switch` | Factory Method |
