Поведінкові патерни відповідають на запитання **"як об'єкти взаємодіють та розподіляють обов'язки"** — як підписуватися на події, як змінювати алгоритми на льоту, як організувати ланцюги обробки, як зберігати стан і відмінювати дії.

## Перелік патернів

| Патерн | Завдання у ЛР4 | Ключова ідея |
|---|---|---|
| Observer            | YouTube-сповіщення           | Підписка / нотифікація |
| Strategy            | Способи оплати кошика        | Алгоритм як об'єкт |
| State               | Робот-пилосос                | Поведінка змінюється зі станом |
| Memento             | Ctrl+Z у редакторі           | Знімок стану без втрати інкапсуляції |
| Chain of Responsibility | Чат-бот → L1 → Tech       | Ланцюг обробників |
| Template Method     | Онлайн-курси                 | Фіксований алгоритм + варіативні кроки |
| Command             | Документообіг з undo         | Дія як об'єкт |
| Iterator            | HRM-система                  | Уніфікований обхід колекцій |
| Mediator            | Авіадиспетчер                | Централізована координація |
| Visitor             | Аудит фінансових документів  | Операції окремо від елементів |

## 1. Observer — YouTube

**Сценарій**: канал публікує відео, всі підписники отримують сповіщення.

**Структура**:
```
YouTubeChannel (Subject)
   └─ _subscribers: List<ISubscriber>
   └─ UploadVideo() → foreach sub: sub.OnNewVideo()

EmailSubscriber, PushSubscriber : ISubscriber
```

**Чому Observer**: автор НЕ знає підписників поіменно. Підписки додаються/прибираються динамічно. Це "publisher-subscriber" або "event-driven" дизайн на найпростішому рівні.

**Зв'язок**: `INotifyPropertyChanged` у .NET — це Observer. `addEventListener` у JavaScript — теж.

**Принципи**: OCP (новий тип підписника = новий клас), LSP (підписники взаємозамінні).

## 2. Strategy — Способи оплати

**Сценарій**: кошик підтримує PayPal / картку / крипту без `switch`.

**Структура**:
```
IPaymentStrategy { Pay(amount); }
   ├─ PayPalStrategy
   ├─ CreditCardStrategy
   └─ CryptoStrategy

ShoppingCart.Checkout(IPaymentStrategy) → strategy.Pay(Total);
```

**Strategy vs State**: схожі за структурою, але:
- **Strategy** обирає клієнт ЗОВНІ.
- **State** обирає сам об'єкт всередині.

**Принципи**: OCP, SRP, заміна `switch` на поліморфізм (антипатерн "type code → strategy").

## 3. State — Робот-пилосос

**Сценарій**: робот діє по-різному залежно від батареї/застрягання — без `if-else` у методі `Tick()`.

**Стани**:
```
CleaningState     → батарея≤20 → SeekingBaseState
                  → stuck=true → StuckState
SeekingBaseState  → батарея≤5 → CleaningState (зарядка)
                  → stuck=true → StuckState
StuckState        → !stuck     → CleaningState
```

**Що змінюється з кожним станом**:
- `Tick()` — основна поведінка.
- Умови переходу — інкапсульовані у стані.

**State vs Strategy**: технічно близькі, але State має значно більше зв'язків між класами стану (вони знають один одного, щоб робити переходи).

**Принципи**: SRP (кожен стан — окремий клас), OCP (новий стан = новий клас).

## 4. Memento — Текстовий редактор з Ctrl+Z

**Сценарій**: undo без порушення інкапсуляції.

**Три ролі**:
- **Originator** — `TextEditor`. Створює і відновлює знімки.
- **Memento** — `EditorMemento`. Immutable знімок з `internal` полями.
- **Caretaker** — `History`. Зберігає memento, але НЕ читає його полів.

**Ключова деталь**:
```csharp
public sealed class EditorMemento {
    internal string Text  { get; }                   // ← internal!
    internal string Color { get; }
}
```

`internal` означає: видно лише в межах того ж збору (assembly). Тобто `TextEditor` (у тому ж assembly) може читати поля при `Restore()`, а зовнішній код — ні. У C++ цю роль грає `friend class`.

**Принципи**: SRP (Memento — тільки знімок, History — тільки історія), Encapsulation preserved.

## 5. Chain of Responsibility — Підтримка

**Сценарій**: бот → L1 → tech. Якщо хтось у ланцюзі впорався — обробив; ні — передав далі.

**Структура**:
```csharp
abstract class SupportHandler {
    private SupportHandler? _next;
    public SupportHandler SetNext(SupportHandler next) { _next = next; return next; }
    public void Handle(req) {
        if (CanHandle(req)) Process(req);
        else _next?.Handle(req);
    }
}
```

**Fluent setup**: `bot.SetNext(l1).SetNext(tech);` — кожен `SetNext` повертає переданий обробник, що дозволяє ланцюжок.

**Граничний випадок**: дойти до кінця ланцюга без обробки — це не помилка, а легітимний результат. У моїй реалізації це логується ("ніхто не зміг").

**Принципи**: SRP (кожен handler — окремий клас), OCP (новий handler = новий клас).

## 6. Template Method — Онлайн-курси

**Сценарій**: усі курси проходяться в однаковій послідовності (лекції→практика→тести→захист), але деякі кроки специфічні.

**Структура**:
```csharp
abstract class OnlineCourse {
    public void Complete() {                         // template method
        WatchLectures();                             // спільний
        DoExercises();                               // спільний
        ExtraVerificationStep();                     // hook — підкласи переозначають
        TakeQuiz();                                  // спільний
        DefendFinalProject();                        // abstract — обов'язково
    }
    protected virtual void ExtraVerificationStep() { } // hook
    protected abstract void DefendFinalProject();
}
```

**Hook vs abstract**:
- `virtual` (hook) — порожня базова реалізація, переозначення опціональне.
- `abstract` — підклас МУСИТЬ реалізувати.

**"Hollywood Principle"**: "Don't call us, we'll call you" — підклас не керує потоком, а лише надає окремі кроки.

**Принципи**: OCP (новий тип курсу = новий підклас), DIP (батько визначає алгоритм через абстракції).

## 7. Command — Документообіг

**Сценарій**: створення, оновлення, видалення документів. Кожна дія повинна підтримувати undo.

**Структура**:
```csharp
interface IDocumentCommand { void Execute(); void Undo(); }

class CreateDocumentCommand : IDocumentCommand { ... }
class UpdateDocumentCommand : IDocumentCommand { ... }
class DeleteDocumentCommand : IDocumentCommand { ... }

class DocumentCommandInvoker {
    private Stack<IDocumentCommand> _executed;
    public void Execute(cmd) { cmd.Execute(); _executed.Push(cmd); }
    public void UndoLast() { _executed.Pop().Undo(); }
}
```

**Ключ до undo**: команда сама зберігає те, що потрібно для відкоту. `DeleteCommand` зберігає бекап вмісту; `UpdateCommand` зберігає старий вміст; `CreateCommand` запам'ятовує ID створеного.

**Розширення**:
- Черга команд → асинхронне виконання.
- Логування → audit trail.
- Серіалізація → виконання на іншому сервері.

**Принципи**: SRP (одна команда — одна дія), OCP (нова дія = нова команда).

## 8. Iterator — HRM

**Сценарій**: компанія має співробітників у відділах (List), проєктах (Dictionary), адмін-групах (List). Клієнт обходить усіх однаково.

**Сила .NET**: `IEnumerable<T>` + `yield return` роблять Iterator майже непомітним:

```csharp
public IEnumerator<Employee> GetEnumerator() {
    foreach (var role in _byRole.Keys)
        foreach (var e in _byRole[role])
            yield return e;                          // ← state machine під капотом
}
```

**Composite iterator**: `CompanyEmployees` приймає кілька джерел і обходить усі по черзі — також через `yield`.

**Спеціалізовані ітератори**:
```csharp
public static IEnumerable<Employee> OnlyActive(source) {
    foreach (var e in source) if (e.Active) yield return e;
}
```
Це LINQ під капотом — `source.Where(e => e.Active)`.

**Принципи**: SRP (структура колекції + спосіб обходу — різні відповідальності), OCP.

## 9. Mediator — Авіадиспетчер

**Сценарій**: літаки, диспетчер, смуга, наземні служби. N×N зв'язків → центральний посередник.

**До**:
```
Aircraft ←→ Dispatcher
Aircraft ←→ RunwayControl
Aircraft ←→ GroundServices
Dispatcher ←→ RunwayControl
Dispatcher ←→ GroundServices
RunwayControl ←→ GroundServices
```
6 пар зв'язків для 4 учасників.

**Після (Mediator)**:
```
       AirportTowerMediator
      ╱     ╲     ╱     ╲
Aircraft  Dispatcher  Runway  Ground
```
Кожен знає тільки посередника.

**Тонкий момент**: у моїй реалізації Mediator робить broadcast (всі отримують усі повідомлення, але реагують лише на свої). Альтернатива — targeted routing (диспетчер знає, кому передати).

**Mediator vs Observer**: схожі ідеї, але Mediator має активну логіку у посереднику, а Observer — просто шину сповіщень.

**Принципи**: зменшення зв'язності, SRP.

## 10. Visitor — Аудит фінансових документів

**Сценарій**: документи (Invoice, Contract, TaxReport) — стабільна структура. Операції (Total, Risk, Export) — змінюються часто.

**Чому НЕ методи в документах**:
```csharp
class Invoice {
    decimal CalculateTotal() { ... }                 // ← операція 1
    bool    AuditRisk()      { ... }                 // ← операція 2
    string  ExportXml()      { ... }                 // ← операція 3
    string  ExportJson()     { ... }                 // ← операція 4
    // ... + 10 нових кожен квартал
}
```
Клас Invoice розростається. Нова операція = редагування ВСІХ типів документів.

**З Visitor**:
```csharp
interface IFinancialVisitor {
    void Visit(Invoice doc);
    void Visit(Contract doc);
    void Visit(TaxReport doc);
}

class Invoice : FinancialDocument {
    public override void Accept(IFinancialVisitor v) => v.Visit(this);  // ← double dispatch
}

class TotalCostVisitor   : IFinancialVisitor { ... }
class RiskAuditVisitor   : IFinancialVisitor { ... }
class ExportVisitor      : IFinancialVisitor { ... }
class ProfitabilityVisitor : IFinancialVisitor { ... }  // ← нова операція — БЕЗ зміни Invoice
```

**Double dispatch**: `Accept(visitor)` викликає `visitor.Visit(this)` — фактичний тип `this` обирає правильний перевантажений метод.

**Платежа за Visitor**: додавання нового ТИПУ документа (`BankStatement`) вимагає змін у ВСІХ візиторах. Vis­itor — для випадків, коли структура стабільна, а операції — мінливі.

**Принципи**: OCP по виміру операцій, SRP (одна операція — один візитор).

## Розподіл у проєкті finance-app (курсова)

Хоча Node.js — не "класичне ООП", деякі патерни легко впізнати:

- **Strategy**: формати експорту (CSV/PDF/JSON) — одна стратегія на формат.
- **Observer**: WebSocket для оновлення дашборду.
- **Command**: REST-ендпоінти POST/PUT/DELETE — кожен як команда.
- **Iterator**: `for...of` по транзакціях з пагінацією.
- **Memento**: рідкісний — на майбутнє, якщо буду робити undo транзакцій.

## Шпаргалка

| Сценарій | Патерн |
|---|---|
| 1 → багато підписників | Observer |
| Поміняти алгоритм на льоту | Strategy |
| Поведінка залежить від внутрішнього стану | State |
| Зберегти/відновити стан | Memento |
| Ланцюг обробників | Chain of Responsibility |
| Алгоритм фіксований, кроки змінюються | Template Method |
| Дія + undo + черга | Command |
| Обійти різні колекції однаково | Iterator |
| Колеги координуються через хаб | Mediator |
| Додати операцію без зміни класів | Visitor |
