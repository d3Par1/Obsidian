## 1. Навіщо SOLID?

**Принципи SOLID — це п'ять основних рекомендацій**, які покращують дизайн програмного забезпечення, роблячи код більш зручним у супроводі та масштабованим.

Принципи SOLID допомагають покращити **слабкий зв'язок** (loose coupling). Слабкий зв'язок означає, що група класів менш залежна один від одного.

**Слабкий зв'язок:**
- робить код більш багаторазовим,
- робить код зручним у супроводі,
- робить код гнучким і стабільним,
- мінімізує зміни у коді, коли потрібні зміни в іншому коді.

**Що дає кожен принцип:**

| Принцип | Що дає |
|---|---|
| **S** — Single Responsibility | Легша підтримка |
| **O** — Open / Closed | Легке додавання функціоналу |
| **L** — Liskov Substitution | Безпечні заміни |
| **I** — Interface Segregation | Фокусовані, модульні інтерфейси |
| **D** — Dependency Inversion | Розділений, зручний для тестування дизайн |

## 2. SOLID — детальний розбір

### S — Принцип єдиної відповідальності (SRP)

> **Кожен клас або модуль має бути відповідальним лише за одну конкретну задачу**, отже, мати лише одну причину для зміни.

**Ключове поняття — `cohesion` (зв'язність/узгодженість).** Чим більший зв'язок між компонентами всередині класу, тим більше програма відповідає принципу єдиної відповідальності.

**🎯 У моїх лабах:**
- **ЛР1 завдання 1** — `Client.Add()` робив 4 речі (валідація, БД, email, оркестрація). Розділено на `Client`, `ClientValidator`, `ClientRepository`, `EmailService`, `ClientService`. → [[SOLID Принципи#S — Single Responsibility Principle|деталі]]
- Опосередковано: кожен патерн ЛР2/3/4 структурно реалізує SRP (один клас = одна роль у патерні).

### O — Принцип відкритості/закритості (OCP)

> **Програмні сутності повинні бути відкритими для розширення, але закритими для модифікації.**

Тобто: можна додавати новий функціонал, але без зміни вже працюючого коду.

**🎯 У моїх лабах:**
- **ЛР1 завдання 3** — `CarRental.Rent()` з `if (rentType == ...)`. Виправлено через Strategy. → [[SOLID Принципи#O — Open/Closed Principle|деталі]]
- **ЛР2 завдання 5** — Factory Method для звітів (новий формат = новий клас).
- **ЛР4 завдання 2** — Strategy для оплати (новий метод оплати = новий клас).

### L — Принцип заміщення Ліскова (LSP)

> **Має бути можливість замість базового типу підставити будь-який його підтип** без додаткових змін в коді.

**Оригінальне формулювання Барбари Лисков (1988):**
> «Якщо для кожного об'єкта `o₁` типу `S` існує об'єкт `o₂` типу `T`, такий, що з будь-якої програми `P`, визначеної в термінах `T`, поведінка `P` не змінюється при заміні `o₂` на `o₁`, то `S` є підтипом `T`.»

LSP стосується трьох речей у контракті базового класу:
- **Передумови (Preconditions)** — підклас не може ПОСИЛЮВАТИ.
- **Постумови (Postconditions)** — підклас не може ПОСЛАБЛЮВАТИ.
- **Інваріанти (Invariants)** — підклас повинен ЗБЕРІГАТИ.

**🎯 У моїх лабах:**
- **ЛР1 завдання 5** — `Square : Rectangle` ламав інваріант (площа = ширина × висота). Виправлено через спільний `IShape`. → [[SOLID Принципи#L — Liskov Substitution Principle|деталі]]
- **ЛР4 Шаблонний метод** — потенційне джерело порушення LSP (підкласи можуть змінити очікувану поведінку кроку).

### I — Принцип розділення інтерфейсів (ISP)

> **Клієнти не повинні залежати від методів, якими не користуються.**

**Техніки виявлення порушення ISP:**
- Занадто великі інтерфейси (10+ методів — майже завжди порушено).
- Компоненти в інтерфейсах слабо узгоджені (перегукується із SRP).
- Методи без реалізації (перегукується з LSP — `NotImplementedException`).

**🎯 У моїх лабах:**
- **ЛР1 завдання 4** — `IEnroll` змушував `ProductEnroll` кидати `NotImplementedException` для `SendEmail`/`SendSMS`. Розділено на 4 спеціалізовані інтерфейси. → [[SOLID Принципи#I — Interface Segregation Principle|деталі]]
- **ЛР3 Adapter** — `IPaymentProvider` має лише 3 методи (`GetAmount`, `GetCurrency`, `GetStatus`) — мінімальний цільовий інтерфейс.

### D — Принцип інверсії залежностей (DIP)

> **Модулі верхнього рівня не повинні залежати від модулів нижнього рівня. І ті, й інші повинні залежати від абстракцій.**
> **Абстракції не повинні залежати від деталей. Деталі мають залежати від абстракцій.**

**🎯 У моїх лабах:**
- **ЛР2 Abstract Factory** — `Application` залежить від `IUIFactory`, а не від `DarkThemeFactory` напряму.
- **ЛР3 Bridge** — `AnalyticsView` залежить від `IRenderer`, а не від конкретного `WebDashboardRenderer`.
- **ЛР4 Strategy** — `ShoppingCart` залежить від `IPaymentStrategy`, а не від `PayPalStrategy`.

## 3. Породжувальні патерни (5 GoF)

> **Породжувальні патерни** абстрагують процес створення екземплярів, допомагають зробити систему незалежною від того, як її об'єкти створюються, компонуються та представляються.

Перелік: Фабричний метод, Абстрактна фабрика, Будівельник, Прототип, Одинак.

### 3.1 Фабричний метод (Factory Method)

**Ідея**: визначити інтерфейс для створення об'єкта, а вибір конкретного класу залишити підкласам.

#### Застосування
- Коли типи і залежності об'єктів, з якими повинен працювати ваш код, **невідомі заздалегідь**.
- Коли ви хочете надати користувачам можливість **розширювати частини вашого фреймворку** чи бібліотеки.
- Коли ви хочете **зекономити системні ресурси**, повторно використовуючи вже створені об'єкти, замість породження нових.

#### Переваги
- Позбавляє клас від прив'язки до конкретних класів продуктів.
- Виділяє код виробництва продуктів в одне місце, спрощуючи підтримку коду.
- Спрощує додавання нових продуктів до програми.
- Реалізує принцип відкритості/закритості (OCP).

#### Недоліки
- Може призвести до створення **великих паралельних ієрархій класів** — для кожного класу продукту потрібен власний підклас творця.

#### Відносини з іншими патернами
- **Багато архітектур починаються з Factory Method** (простіший) і **еволюціонують у Абстрактну фабрику / Прототип / Будівельник** (гнучкіші, але складніші).
- Класи **Абстрактної фабрики** найчастіше реалізуються за допомогою Фабричного методу.
- Factory Method можна використовувати разом з **Ітератором** — підкласи колекцій створюють потрібні їм ітератори.
- **Прототип не спирається на спадкування**, але потребує складної ініціалізації; Factory Method, навпаки, побудований на спадкуванні, але не вимагає складної ініціалізації.
- Factory Method можна розглядати як окремий випадок **Шаблонного методу**.

#### 🎯 У моїх лабах
**ЛР2 завдання 5** — експорт звітів у PDF/Excel/HTML/JSON. `ReportFactoryResolver.GetFactory(format)` повертає відповідний підклас `ReportFactory`. → [[Породжувальні патерни#5. Factory Method — Експорт звітів|код]]

### 3.2 Абстрактна фабрика (Abstract Factory)

**Ідея**: створювати **сімейства** пов'язаних об'єктів без прив'язки до їхніх конкретних класів.

#### Застосування
- Коли бізнес-логіка програми повинна **працювати з різними видами пов'язаних один з одним продуктів**, незалежно від конкретних класів.
- Коли в програмі вже використовується Factory Method, але чергові зміни **передбачають введення нових типів продуктів**.

#### Переваги
- **Гарантує поєднання створюваних продуктів** (сумісність всередині сімейства).
- Звільняє клієнтський код від прив'язки до конкретних класів продукту.
- Виділяє код виробництва продуктів в одне місце.
- Спрощує додавання нових продуктів до програми.
- Реалізує принцип відкритості/закритості.

#### Недоліки
- Ускладнює код програми внаслідок введення великої кількості додаткових класів.
- Вимагає наявності всіх типів продукту в кожній варіації (якщо додав новий продукт — мусиш додати у ВСІ фабрики).

#### Відносини з іншими патернами
- Абстрактна фабрика часто реалізується через **Фабричний метод**, але може бути побудована і на **Прототипі**.
- Може бути використана **замість Фасаду**, щоб приховати платформо-залежні класи.
- Може працювати спільно з **Мостом**: фабрика визначає типи створюваних абстракцій та реалізацій.
- Абстрактна фабрика, **Будівельник** та **Прототип** можуть реалізовуватися за допомогою **Одинака**.
- **Будівельник vs Abstract Factory**: Builder будує крок за кроком і повертає продукт В КІНЦІ; Abstract Factory повертає продукт ОДРАЗУ.

#### 🎯 У моїх лабах
**ЛР2 завдання 2** — UI-теми. `IUIFactory` створює сімейство `IButton + ITextField + IDropdown`. Заміна `DarkThemeFactory` на `LightThemeFactory` змінює ВСЮ тему. → [[Породжувальні патерни#2. Abstract Factory — UI-теми|код]]

### 3.3 Будівельник (Builder)

**Ідея**: відокремити побудову складного об'єкта від його представлення, щоб один і той самий процес побудови міг створювати різні представлення.

#### Застосування
- Коли ви хочете **позбутися "телескопічного конструктора"** (конструктор з 10 параметрами, частина яких опціональна).
- Коли ваш код повинен створювати **різні уявлення одного об'єкта** (наприклад, дерев'яні та залізобетонні будинки).
- Коли вам потрібно збирати **складні об'єкти**, наприклад дерева Компонувальника.

#### Переваги
- Дозволяє **створювати продукти покроково**.
- Дозволяє використовувати один і той самий код для створення різноманітних продуктів.
- **Ізолює складний код конструювання** продукту від його головної бізнес-логіки.

#### Недоліки
- Ускладнює код програми за рахунок додаткових класів.
- Клієнт буде прив'язаний до конкретних класів будівельників, бо в інтерфейсі будівельника може не бути методу отримання результату.

#### Відносини з іншими патернами
- Архітектури часто **еволюціонують від Factory Method до Builder / Abstract Factory / Prototype**.
- **Builder vs Abstract Factory**: Builder фокусується на ПОКРОКОВІЙ побудові СКЛАДНИХ об'єктів. Abstract Factory створює СІМЕЙСТВА пов'язаних продуктів. Builder повертає продукт після ВСІХ кроків; Abstract Factory повертає продукт ОДРАЗУ.
- Builder дозволяє покроково конструювати **дерево Компонувальника**.
- Builder може бути побудований у вигляді **Мосту**: директор — абстракція, будівельники — реалізації.
- Builder + Abstract Factory + Prototype можуть реалізовуватися як **Singleton**.

#### 🎯 У моїх лабах
**ЛР2 завдання 1** — створення персонажа гри. `CharacterBuilder` з fluent API `.SetName().SetClass().Build()`. `CharacterDirector` зберігає рецепти "стандартний воїн" і "стандартний маг". → [[Породжувальні патерни#1. Builder — Творець персонажа|код]]

### 3.4 Прототип (Prototype)

**Ідея**: створювати нові об'єкти **копіюванням існуючих**, без прив'язки до їхніх конкретних класів.

**Проблема, яку вирішує Прототип:** не кожен об'єкт можна скопіювати "ззовні" — частина його стану приватна. А код, що копіює "ззовні", буде залежним від конкретних класів об'єктів. Тому копіювання повинне бути обов'язком самого об'єкта — через метод `Clone()`.

**Дві реалізації:**
- **Базова** — кожен прототип знає, як себе клонувати.
- **Зі сховищем прототипів** — окремий реєстр готових прототипів, з якого беруть і клонують.

#### Застосування
- Коли ваш код не повинен залежати від класів об'єктів, призначених для копіювання.
- Коли ви маєте **безліч підкласів, які відрізняються лише початковими значеннями полів** — заміняємо їх одним класом + сховищем прототипів.

#### Переваги
- Дозволяє клонувати об'єкти **без прив'язки до конкретних класів**.
- Менша кількість повторювань коду ініціалізації об'єктів.
- **Прискорює створення** об'єктів (особливо якщо конструктор важкий).
- Альтернатива створенню підкласів під час конструювання складних об'єктів.

#### Недоліки
- **Складно клонувати складові об'єкти**, що мають посилання на інші об'єкти (потрібно вирішувати, deep чи shallow copy).

#### Відносини з іншими патернами
- Архітектури часто еволюціонують від Factory Method до Prototype.
- Якщо **Команду потрібно копіювати** перед вставкою в історію виконаних команд — допоможе Prototype.
- Архітектура на Компонувальниках / Декораторах часто **поліпшується введенням Прототипу** (клонувати складні структури, а не збирати заново).
- **Знімок (Memento) іноді можна замінити Прототипом**, якщо об'єкт простий і не має посилань на зовнішні ресурси.
- Prototype не спирається на спадкування (на відміну від Factory Method), але потребує складної ініціалізації.
- Abstract Factory + Builder + Prototype можуть реалізовуватися як Singleton.

#### 🎯 У моїх лабах
**ЛР2 завдання 4** — клонування орків у RTS. `OrcPrototype.Clone()` копіює індивідуальні поля (позиція, рівень) і передає посиланням спільні (текстура, ефекти) — це вже Flyweight всередині Prototype. → [[Породжувальні патерни#4. Prototype — Клонування орків|код]]

### 3.5 Одинак (Singleton)

**Ідея**: вирішує одразу дві проблеми:
1. **Гарантує наявність єдиного екземпляра** класу.
2. **Надає глобальну точку доступу** до нього.

#### Застосування
- Коли в програмі повинен бути **єдиний екземпляр** якого-небудь класу, доступний усім клієнтам (спільний доступ до бази даних, лог-файла, конфігурації).
- Коли ви хочете мати **більше контролю над глобальними змінними**.

#### Переваги
- Гарантує наявність єдиного екземпляра класу.
- Надає глобальну точку доступу.
- Реалізує **відкладену ініціалізацію** (lazy initialization).

#### Недоліки
- **Порушує принцип єдиного обов'язку класу** (клас і керує своїм життєвим циклом, і виконує бізнес-логіку).
- **Маскує поганий дизайн** — Singleton часто пхають туди, де насправді потрібен DI.
- **Проблеми багатопоточності** (якщо немає правильної синхронізації — будуть два екземпляри).
- Вимагає **постійного створення Mock-об'єктів при юніт-тестуванні** (бо Singleton глобальний — складно ізолювати тести).

#### Часта помилка та її вирішення

**Помилка**: код, що залежить від Singleton, неможливо протестувати без реального екземпляра.

**Варіант вирішення №1 — Wrapper**: обгорнути Singleton в інтерфейс і передавати через параметр.
```csharp
interface ILogService { void Log(string msg); }
class LogServiceWrapper : ILogService {
    public void Log(string msg) => LogService.Instance.Log(msg);
}
// Клієнт залежить від ILogService, не від Singleton
```

**Варіант вирішення №2 — Dependency Injection**: замість Singleton.Instance, реєструвати клас як `Singleton` lifetime у DI-контейнері. Зовнішньо це Singleton, але клієнт отримує його через конструктор.
```csharp
services.AddSingleton<ILogService, LogService>();
class MyService(ILogService logger) { ... }  // тестується mock-ом
```

#### Відносини з іншими патернами
- **Фасад** можна зробити Singleton (зазвичай потрібен один фасад).
- **Легковаговик** нагадує Singleton, але:
  - Можна мати БАГАТО Легковаговиків.
  - Легковаговики незмінні (immutable), Singleton допускає зміну стану.
- Abstract Factory + Builder + Prototype можуть бути Singleton.

#### 🎯 У моїх лабах
**ЛР2 завдання 3** — `LogService` з потокобезпечним double-check locking. → [[Породжувальні патерни#3. Singleton — Сервіс логування|код]]

### 3.6 Трохи термінології (важливо для захисту!)

Якщо викладач питає "що таке фабрика?" — слово вживають по-різному:

- **Фабрика (Factory)** — загальний термін. Може означати:
  - функцію чи метод, що створює об'єкти програми;
  - клас, який створює користувачів системи;
  - статичний метод, який обертає конструктор класу;
  - один із класичних фабричних патернів (Factory Method / Abstract Factory).
- **Створюючий метод (Creating method)** — метод, основна задача якого — створити об'єкт. Не патерн, а просто роль методу.
- **Статичний фабричний метод (Static factory method)** — статичний метод, що повертає екземпляр класу. Може мати інформативну назву (`User.CreateGuest()` vs `new User(isGuest: true)`).
- **Проста фабрика (Simple factory)** — клас, який інкапсулює логіку створення (часто з `switch`). **НЕ є GoF-патерном** — це антипатерн з точки зору OCP. Іноді її помилково називають Factory Method.
- **Фабричний метод (Factory Method)** — GoF-патерн. Базовий клас оголошує метод створення; підкласи перевизначають його.
- **Абстрактна фабрика (Abstract Factory)** — GoF-патерн. Інтерфейс для створення СІМЕЙСТВА об'єктів.

> **На захисті**: якщо плутаєш терміни — кажи "Factory Method" або "Abstract Factory" англійською, це знімає двозначність.

## 4. Структурні патерни (7 GoF)

> **Структурні шаблони** зосереджені на **організації класів та об'єктів** для створення більших, ефективних та зручних у підтримці структур ПЗ.

Перелік: Адаптер, Міст, Компонувальник, Декоратор, Фасад, Легковаговик, Замісник.

### 4.1 Адаптер (Adapter)

**Ідея**: адаптер має інтерфейс, сумісний з одним об'єктом, і **перенаправляє виклики іншому об'єкту** в потрібному форматі.

**Два варіанти:**
- **Адаптер об'єктів** — через композицію (новий клас зберігає посилання на adaptee).
- **Адаптер класів** — через множинне спадкування (можливо в C++; у C# — лише через інтерфейси).

#### Застосування
- Якщо ви хочете **використати сторонній клас**, але його інтерфейс не відповідає решті кодів програми.
- Якщо вам потрібно використати **декілька існуючих підкласів**, але в них не вистачає якоїсь спільної функціональності, а розширити суперклас ви не можете.

#### Переваги
- **Відокремлює та приховує від клієнта** подробиці перетворення різних інтерфейсів.

#### Недоліки
- Ускладнює код програми внаслідок додаткових класів.

#### Відносини з іншими патернами
- **Bridge проектують ЗАЗДАЛЕГІДЬ**, щоб розвивати великі частини програми окремо. **Adapter застосовується ПОСТФАКТУМ**, щоб змусити несумісні класи працювати разом.
- **Adapter vs Decorator vs Proxy** — всі обгортають об'єкт, але:
  - Adapter — інтерфейс **повністю інший**.
  - Proxy — інтерфейс **той самий**.
  - Decorator — інтерфейс **той самий або розширений**, плюс підтримує рекурсивну вкладеність.
- **Fasade vs Adapter**:
  - Facade задає **новий** інтерфейс; Adapter повторно використовує старий.
  - Adapter обгортає **один** клас; Facade обгортає **цілу підсистему**.
  - Adapter дозволяє двом ІСНУЮЧИМ інтерфейсам працювати разом; Facade ВИЗНАЧАЄ новий інтерфейс.
- **Bridge / Strategy / State / (частково Adapter)** мають схожі структури класів — всі побудовані на принципі **композиції** (делегування роботи іншим об'єктам), але вирішують різні проблеми.

#### 🎯 У моїх лабах
**ЛР3 завдання 1** — уніфікація платіжних API (JSON / XML / async crypto). Кожен Adapter реалізує `IPaymentProvider`. `CryptoPaymentAdapter` додатково приховує асинхронність через `.GetAwaiter().GetResult()`. → [[Структурні патерни#1. Adapter — Платіжні провайдери|код]]

### 4.2 Міст (Bridge)

**Ідея**: **розділити одну ієрархію класів на дві**, дозволяючи їм змінюватися незалежно.

#### Застосування
- Якщо ви хочете **розділити монолітний клас**, який містить кілька різних реалізацій якоїсь функціональності (наприклад, клас може працювати з різними системами БД).
- Якщо клас потрібно **розширювати в двох незалежних площинах**.
- Якщо ви хочете мати **можливість змінювати реалізацію під час виконання** програми.

#### Переваги
- Дозволяє будувати **платформо-незалежні програми**.
- Приховує зайві або небезпечні деталі реалізації від клієнтського коду.
- Реалізує принцип відкритості/закритості.

#### Недоліки
- Ускладнює код програми внаслідок додаткових класів.

#### Відносини з іншими патернами
- Bridge проектують ЗАЗДАЛЕГІДЬ, Adapter — ПОСТФАКТУМ (див. Adapter).
- Bridge, Strategy, State, (частково Adapter) — всі побудовані на композиції, але вирішують різні проблеми.
- **Abstract Factory може працювати з Bridge** — фабрика визначає типи створюваних абстракцій та реалізацій.
- **Builder може бути побудований як Bridge**: директор — абстракція, будівельники — реалізації.

#### 🎯 У моїх лабах
**ЛР3 завдання 6** — аналітика × відображення (3 типи даних × 3 рендерери). `AnalyticsView` (Abstraction) і `IRenderer` (Implementor). Замість 9 класів — 6. → [[Структурні патерни#6. Bridge — Аналітика × Відображення|код]]

### 4.3 Компонувальник (Composite)

**Ідея**: дозволяє трактувати **окремі об'єкти і композиції** через **єдиний інтерфейс**.

**Класичний приклад**: Продукт (листок) і Коробка з продуктами (композит). Обидва мають метод `GetPrice()`. Продукт повертає свою ціну, Коробка сумує ціни вмісту.

#### Застосування
- Якщо вам потрібно представити **деревоподібну структуру об'єктів**.
- Якщо клієнти повинні **однаково трактувати прості та складові об'єкти**.

#### Переваги
- Спрощує архітектуру клієнта при роботі зі складним деревом компонентів.
- Полегшує додавання нових видів компонентів (новий листок або новий тип композита).

#### Недоліки
- **Створює занадто загальний дизайн класів** (іноді листок мусить мати методи, що мають сенс лише для композита, або навпаки).

#### Відносини з іншими патернами
- **Builder дозволяє покроково конструювати дерево Компонувальника**.
- **Chain of Responsibility** часто використовують разом з Композитом — запит передається від дочірніх до батьків.
- Ви можете обходити дерево Композита, використовуючи **Iterator**.
- Ви можете виконати дію над усім деревом через **Visitor**.
- **Composite + Flyweight** — спільні гілки дерева економлять пам'ять.
- **Composite + Decorator** — обидва на рекурсивній вкладеності. Архітектура з обома часто **поліпшується введенням Prototype** (клонувати готові структури).

#### 🎯 У моїх лабах
**ЛР3 завдання 4** — CRM-модулі з ієрархією. `LeafModule` (окремий модуль) і `CompositeModule` (група). `GetTotalPrice()` працює рекурсивно. Каскадне вимкнення гілки. → [[Структурні патерни#4. Composite — CRM-модулі|код]]

### 4.4 Декоратор (Decorator)

**Ідея**: **додавати об'єктам нові обов'язки "на льоту"**, обертаючи їх у спеціальні класи-декоратори.

#### Застосування
- Якщо вам потрібно **додавати об'єктам нові обов'язки "на льоту"**, непомітно для коду, який їх використовує.
- Якщо **не можна розширити обов'язки об'єкта за допомогою спадкування** (фінальний клас, комбінаторний вибух).

#### Переваги
- **Більша гнучкість**, ніж у спадкування.
- Дозволяє додавати обов'язки "на льоту".
- Можна додавати кілька нових обов'язків одразу.
- Дозволяє мати **кілька дрібних об'єктів** замість одного "на всі випадки життя".

#### Недоліки
- **Важко конфігурувати об'єкти**, які загорнуто в декілька обгорток одночасно.
- Велика кількість крихітних класів.

#### Відносини з іншими патернами
- **Decorator vs Adapter vs Proxy**: всі обгортають об'єкт. Decorator РОЗШИРЮЄ інтерфейс або зберігає, плюс рекурсивна вкладеність.
- **Composite + Decorator** — схожі структури класів (обидва рекурсивні). Архітектура з обома часто поліпшується введенням **Prototype**.
- Decorator vs Спадкування — див. також [[Захист Лабораторних#2. Наслідування в C++ та C#|переваги композиції]].

#### 🎯 У моїх лабах
**ЛР3 завдання 5** — SaaS-документи з комбінацією функцій (шифрування + стиснення + водяний знак + аудит). 4 функції → 4 декоратори, а не 16 класів комбінацій. → [[Структурні патерни#5. Decorator — SaaS-документи|код]]

### 4.5 Фасад (Facade)

**Ідея**: надати **простий уніфікований інтерфейс** до складної підсистеми.

#### Застосування
- Якщо вам потрібно надати **простий або урізаний інтерфейс** до складної підсистеми.
- Якщо ви хочете **розкласти підсистему на окремі рівні**.

#### Переваги
- **Ізолює клієнтів** від компонентів складної підсистеми.

#### Недоліки
- **Facade ризикує стати "божественим об'єктом"** (god object), прив'язаним до всіх класів програми.

#### Відносини з іншими патернами
- **Abstract Factory може бути використана замість Facade**, щоб приховати платформо-залежні класи.
- **Facade vs Adapter** — див. секцію Adapter.
- Facade можна зробити **Singleton** (зазвичай один фасад).

#### 🎯 У моїх лабах
**ЛР3 завдання 2** — оформлення замовлення з 6 підсистемами (Pricing, Discount, Inventory, Payment, Shipping, Notification). Один виклик `OrderFacade.PlaceOrder()`. → [[Структурні патерни#2. Facade — Оформлення замовлення|код]]

### 4.6 Легковаговик (Flyweight)

**Ідея**: **розділити стан об'єкта на спільний (intrinsic)** і **унікальний (extrinsic)**, щоб мільйони об'єктів могли ділити intrinsic-частину.

#### Застосування
- Якщо **не вистачає оперативної пам'яті** для підтримки всіх потрібних об'єктів.

#### Переваги
- **Заощаджує оперативну пам'ять** (іноді на порядки).

#### Недоліки
- Витрачає процесорний час на пошук/обчислення контексту (extrinsic state передається ззовні щоразу).
- Ускладнює код програми через введення додаткових класів (Factory + два стани).

#### Відносини з іншими патернами
- **Flyweight vs Singleton**: обидва часто є одинаком, але:
  - Можна мати БАГАТО Flyweight-ів.
  - Flyweight-и **незмінні** (immutable); Singleton допускає зміну стану.
- **Composite + Flyweight** — спільні гілки дерева економлять пам'ять.

#### 🎯 У моїх лабах
**ЛР3 завдання 7** — Minecraft-світ із 100 000+ блоків. `BlockType` (intrinsic — текстура, модель) у пулі `BlockTypeFactory`. `WorldBlock` (extrinsic — позиція, поточна міцність). Замість 100 000 копій важких текстур — 5 типів. → [[Структурні патерни#7. Flyweight — Minecraft-блоки|код]]

### 4.7 Замісник (Proxy)

**Ідея**: **підставити замість справжнього об'єкта замісника**, який контролює доступ до нього.

#### Застосування (5 типів Proxy)
- **Лінива ініціалізація (Virtual Proxy)** — коли є важкий об'єкт, що завантажує дані з файлової системи або БД.
- **Захист доступу (Protection Proxy)** — коли в програмі різні типи користувачів, і потрібно захистити об'єкт від неавторизованого доступу.
- **Локальний запуск сервісу (Remote Proxy)** — коли справжній сервісний об'єкт на віддаленому сервері.
- **Логування запитів (Logging Proxy)** — коли потрібно зберігати історію звернень.
- **Кешування об'єктів (Caching Proxy / "smart reference")** — кешує результати запитів клієнтів.

#### Переваги
- Дозволяє **контролювати сервісний об'єкт непомітно** для клієнта.
- Може **працювати, навіть якщо сервісний об'єкт ще не створено** (Virtual Proxy).
- Може **контролювати життєвий цикл** службового об'єкта.

#### Недоліки
- Ускладнює код програми через введення додаткових класів.
- Збільшує час отримання відклику від сервісу (зайва ланка).

#### Відносини з іншими патернами
- **Proxy vs Decorator vs Adapter** — всі обгортають. Proxy зберігає інтерфейс і **контролює доступ**; Decorator РОЗШИРЮЄ; Adapter — інтерфейс інший.

#### 🎯 У моїх лабах
**ЛР3 завдання 3** — стрімінг фільмів. `MovieProxy` поєднує 3 ролі: Virtual Proxy (lazy `Load()`), Cache Proxy (`_cached`), Protection Proxy (`_userHasPremium`). → [[Структурні патерни#3. Proxy — Lazy завантаження фільмів|код]]

## 5. Поведінкові патерни (10 GoF)

> **Поведінкові патерни** стосуються **взаємодії та комунікації об'єктів** — як вони обмінюються повідомленнями, розподіляють відповідальності, координують роботу.

Перелік: Ланцюжок обов'язків, Команда, Ітератор, Посередник, Знімок, Спостерігач, Стан, Стратегія, Шаблонний метод, Відвідувач.

### 5.1 Ланцюжок обов'язків (Chain of Responsibility)

**Ідея**: **передавати запит послідовно по ланцюжку обробників**, поки хтось не обробить.

#### Застосування
- Якщо програма має **обробляти різноманітні запити багатьма способами**, але заздалегідь невідомо, які запити надходитимуть і які обробники для них знадобляться.
- Якщо важливо, щоб **обробники виконувалися у суворому порядку**.
- Якщо набір об'єктів, здатних обробити запит, повинен задаватися **динамічно**.

#### Переваги
- Зменшує залежність між клієнтом та обробниками.
- Реалізує **принцип єдиного обов'язку** (кожен handler має одну роль).
- Реалізує **принцип відкритості/закритості** (новий handler = новий клас).

#### Недоліки
- **Запит може залишитися ніким не опрацьованим** (потрібно явно обробляти цей випадок).

#### Відносини з іншими патернами
- Chain of Responsibility часто використовують разом з **Композитом** — запит передається від дочірніх до батьків.
- **Приклад з лекцій**: ланцюжок обов'язків відповідає за показ контекстної допомоги для активних елементів UI.

#### 🎯 У моїх лабах
**ЛР4 завдання 5** — підтримка користувачів. `bot.SetNext(l1).SetNext(tech)`. Якщо обробник не може — передає далі. Фінал ланцюга = логування "ніхто не зміг". → [[Поведінкові патерни#5. Chain of Responsibility — Підтримка|код]]

### 5.2 Команда (Command)

**Ідея**: **обернути запит в об'єкт**, щоб параметризувати клієнтів, ставити запити в чергу, логувати та підтримувати undo.

#### Застосування
- Якщо ви хочете **параметризувати об'єкти виконуваною дією**.
- Якщо ви хочете **поставити операції в чергу**, виконувати їх за розкладом або передавати мережею.
- Якщо вам потрібна **операція скасування** (undo).

#### Переваги
- Прибирає **пряму залежність** між об'єктами, що викликають операції, та об'єктами, які їх виконують.
- Дозволяє **просте скасування і повтор** операцій.
- Дозволяє реалізувати **відкладений запуск** операцій.
- Дозволяє збирати **складні команди з простих** (Composite Command).
- Реалізує принцип відкритості/закритості.

#### Недоліки
- **Ускладнює код програми** через велику кількість додаткових класів (одна команда = один клас).

#### Відносини з іншими патернами
- Якщо Команду треба копіювати перед вставкою в історію — допоможе **Prototype**.

#### 🎯 У моїх лабах
**ЛР4 завдання 7** — документообіг з undo. `IDocumentCommand { Execute() + Undo() }`. `Create/Update/Delete` зберігають у собі бекап для відкоту. → [[Поведінкові патерни#7. Command — Документообіг|код]]

### 5.3 Ітератор (Iterator)

**Ідея**: **уніфіковано обходити елементи колекції** без розкриття її внутрішньої структури.

**Ключова теза**: одну і ту саму колекцію можна обходити різними способами (вперед/назад, з фільтром, у певному порядку).

#### Застосування
- Якщо у вас є **складна структура даних**, і ви хочете приховати від клієнта деталі її реалізації.
- Якщо вам потрібно мати **кілька варіантів обходу** однієї і тієї ж структури.
- Якщо вам потрібен **єдиний інтерфейс обходу різних структур** даних.

#### Переваги
- **Спрощує класи зберігання даних** (вони не мусять самі реалізовувати обхід).
- Дозволяє реалізувати **різні способи обходу**.
- Дозволяє **одночасно переміщуватися структурою даних у різних напрямках**.

#### Недоліки
- **Невиправданий, якщо можна обійтися простим циклом**.

#### Відносини з іншими патернами
- Iterator часто використовується для обходу дерева **Композита**.
- **Factory Method + Iterator** — підкласи колекцій створюють потрібні їм ітератори.

#### 🎯 У моїх лабах
**ЛР4 завдання 8** — HRM-система. `Department` (List), `ProjectTeam` (Dictionary), `CompanyEmployees` (composite). Усі через `IEnumerable<Employee>` + `yield return`. → [[Поведінкові патерни#8. Iterator — HRM|код]]

### 5.4 Посередник (Mediator)

**Ідея**: **звести зв'язки N×N до зірки** — всі компоненти спілкуються через центральний посередник.

#### Застосування
- Коли вам **складно змінювати деякі класи** через те, що вони мають **величезну кількість хаотичних зв'язків** з іншими.
- Коли ви **не можете повторно використовувати клас**, оскільки він залежить від безлічі інших.
- Коли вам доводиться створювати багато підкласів компонентів, щоб **використовувати одні й ті самі компоненти в різних контекстах**.

#### Переваги
- **Усуває залежності** між компонентами, дозволяючи їх повторно використовувати.
- Спрощує взаємодію між компонентами.
- **Централізує керування** в одному місці.

#### Недоліки
- **Посередник може "роздутися"** і стати god object-ом (як Facade).

#### 🎯 У моїх лабах
**ЛР4 завдання 9** — авіадиспетчер. Літак, диспетчер, смуга, наземні служби — всі через `AirportTowerMediator`. Broadcast routing. → [[Поведінкові патерни#9. Mediator — Авіадиспетчер|код]]

### 5.5 Знімок (Memento / Snapshot)

**Ідея**: **зберегти стан об'єкта** для подальшого відновлення, **не розкриваючи його приватних деталей**.

**Три варіанти реалізації:**
- **Класична на вкладених класах** — `Memento` є nested class всередині `Originator` (доступ до приватних полів).
- **З проміжним порожнім інтерфейсом** — `Memento` реалізує порожній публічний інтерфейс, який бачить Caretaker; повний інтерфейс — приватний.
- **З підвищеним захистом** — `Memento` має `internal`-поля (як у моїй реалізації).

#### Застосування
- Коли вам потрібно **зберігати миттєві знімки стану** об'єкта для подальшого відновлення.
- Коли пряме отримання стану об'єкта **розкриває приватні деталі**, порушуючи інкапсуляцію.

#### Переваги
- **Не порушує інкапсуляцію** вихідного об'єкта.
- **Спрощує структуру вихідного об'єкта** — йому не потрібно зберігати історію.

#### Недоліки
- **Вимагає багато пам'яті**, якщо клієнти дуже часто створюють знімки.
- Може спричинити витрати пам'яті, якщо Caretaker не звільняє застарілі знімки.
- **В деяких мовах (PHP, Python, JS) складно гарантувати**, щоб лише Originator мав доступ до стану Memento (немає `internal`/`friend`).

#### Відносини з іншими патернами
- **Memento можна замінити Prototype**, якщо стан простий і не має посилань.
- **Memento + Command** — для undo-функціональності в команді.

#### 🎯 У моїх лабах
**ЛР4 завдання 4** — текстовий редактор з Ctrl+Z. `EditorMemento` має `internal`-поля. `History` (Caretaker) тримає стек знімків. → [[Поведінкові патерни#4. Memento — Текстовий редактор з Ctrl+Z|код]]

### 5.6 Спостерігач (Observer)

**Ідея**: **один-до-багатьох**. Об'єкт-видавець сповіщає всіх підписників про зміни.

#### Застосування
- Якщо **після зміни стану одного об'єкта потрібно щось зробити в інших**, але ви не знаєте наперед, які саме об'єкти мають відреагувати.
- Якщо одні об'єкти **мають спостерігати за іншими**, але тільки у визначених випадках.

#### Переваги
- **Видавці не залежать від конкретних класів підписників** і навпаки.
- Можна **підписувати і відписувати** одержувачів "на льоту".
- Реалізує принцип відкритості/закритості.

#### Недоліки
- **Підписники сповіщуються у випадковій послідовності** (якщо не задано пріоритет).

#### 🎯 У моїх лабах
**ЛР4 завдання 1** — YouTube. `YouTubeChannel` (Subject) тримає `List<ISubscriber>`. `EmailSubscriber` + `PushSubscriber`. → [[Поведінкові патерни#1. Observer — YouTube|код]]

### 5.7 Стан (State)

**Ідея**: дозволити об'єкту **змінювати свою поведінку** залежно від внутрішнього стану. Виглядає так, ніби об'єкт змінив свій клас.

#### Застосування
- Якщо у вас є об'єкт, **поведінка якого кардинально змінюється** залежно від внутрішнього стану, причому типів станів багато і їх код часто змінюється.
- Якщо код класу **містить безліч великих, схожих один на одного умовних операторів**, які вибирають поведінку.
- Якщо ви свідомо **використовуєте табличну машину станів**, але змушені миритися з дублюванням коду.

#### Переваги
- **Позбавляє від безлічі великих умовних операторів** машини станів.
- **Концентрує в одному місці код**, пов'язаний з певним станом.
- **Спрощує код контексту** (Context не містить логіки переходів — вона у станах).

#### Недоліки
- Може **невиправдано ускладнити код**, якщо станів мало і вони рідко змінюються.

#### Відносини з іншими патернами
- **State vs Strategy** — структурно схожі, але:
  - Strategy обирає клієнт ЗОВНІ.
  - State обирає сам об'єкт ВСЕРЕДИНІ.
- State / Strategy / Bridge / Adapter — побудовані на композиції.

#### 🎯 У моїх лабах
**ЛР4 завдання 3** — робот-пилосос. Стани: `CleaningState`, `SeekingBaseState`, `StuckState`. У `VacuumRobot.Tick()` нуль `if` — лише `_state.Tick(this)`. Переходи всередині станів. → [[Поведінкові патерни#3. State — Робот-пилосос|код]]

### 5.8 Стратегія (Strategy)

**Ідея**: **визначити сімейство алгоритмів**, інкапсулювати кожен, і зробити їх взаємозамінними.

#### Застосування
- Якщо вам потрібно **використовувати різні варіації якогось алгоритму** всередині одного об'єкта.
- Якщо у вас є **безліч схожих класів**, які відрізняються лише деякою поведінкою.
- Якщо ви **не хочете оголювати деталі реалізації** алгоритмів для інших класів.
- Якщо різні варіації алгоритмів **реалізовано як розлогий умовний оператор** — кожна гілка є варіацією.

#### Переваги
- **Гаряча заміна алгоритмів на льоту**.
- Ізолює код і дані алгоритмів від інших класів.
- **Заміна спадкування делегуванням**.
- Реалізує принцип відкритості/закритості.

#### Недоліки
- Ускладнює програму через додаткові класи.
- **Клієнт повинен знати, в чому полягає різниця між стратегіями**, щоб вибрати потрібну.

#### Відносини з іншими патернами
- **Strategy vs State** — див. State.
- Strategy / State / Bridge / Adapter — на композиції.

#### 🎯 У моїх лабах
**ЛР4 завдання 2** — оплата кошика. `IPaymentStrategy { Pay(amount); }`. `PayPalStrategy / CreditCardStrategy / CryptoStrategy`. Нуль `switch`. → [[Поведінкові патерни#2. Strategy — Способи оплати|код]]

### 5.9 Шаблонний метод (Template Method)

**Ідея**: **визначити скелет алгоритму** в базовому класі, **передаючи реалізацію деяких кроків підкласам**.

#### Застосування
- Якщо підкласи повинні **розширювати базовий алгоритм, не змінюючи його структури**.
- Якщо у вас є **кілька класів, які роблять одне й те саме з незначними відмінностями** — редагування одного вимагає змін у всіх.

#### Переваги
- **Полегшує повторне використання коду**.

#### Недоліки
- Ви **жорстко обмежені скелетом** існуючого алгоритму.
- **Можете порушити принцип Барбари Ліскової**, змінюючи базову поведінку через підклас (якщо переозначити крок з іншою семантикою).
- У міру зростання кількості кроків Шаблонний метод стає **занадто складно підтримувати**.

#### Відносини з іншими патернами
- **Factory Method** часто є частиною Template Method (`CreateProduct()` — крок).
- Strategy — альтернатива: композиція замість успадкування.

#### 🎯 У моїх лабах
**ЛР4 завдання 6** — онлайн-курси. `OnlineCourse.Complete()` — template method. `ExtraVerificationStep()` — hook. `DefendFinalProject()` — abstract. → [[Поведінкові патерни#6. Template Method — Онлайн-курси|код]]

### 5.10 Відвідувач (Visitor)

**Ідея**: **винести операції з класів в окремі візитори**, щоб додавати нові операції без зміни класів.

#### Застосування
- Якщо вам потрібно **виконати якусь операцію над усіма елементами складної структури** (наприклад, дерева).
- Якщо над об'єктами структури потрібно виконувати **не пов'язані між собою операції**, але ви не хочете "засмічувати" класи цими операціями.
- Якщо нова поведінка **має сенс тільки для деяких класів** з існуючої ієрархії.

#### Переваги
- **Спрощує додавання операцій**, працюючих зі складними структурами.
- **Об'єднує споріднені операції** в одному класі.
- Visitor може **накопичувати стан** при обході структури.

#### Недоліки
- **Невиправданий, якщо ієрархія елементів часто змінюється** — кожна зміна вимагає правки ВСІХ візиторів.
- Може **призвести до порушення інкапсуляції** елементів (Element змушений виставити поля назовні для Visitor).

#### Відносини з іншими патернами
- Visitor + **Composite** — Visitor виконує дію над усім деревом.
- Visitor + **Iterator** — поєднують обхід і операцію.

#### 🎯 У моїх лабах
**ЛР4 завдання 10** — аудит фінансових документів. `Invoice / Contract / TaxReport` — Elements. `TotalCostVisitor / RiskAuditVisitor / ExportVisitor` — Visitors. Double dispatch через `Accept(v) → v.Visit(this)`. → [[Поведінкові патерни#10. Visitor — Аудит фінансових документів|код]]

## 7. Як використовувати цей файл

**Перед захистом:**
1. Прочитати один раз повністю — це база.
2. Для кожного з 22 патернів запам'ятати ОДНУ ключову тезу з "Застосування".
3. Запам'ятати ОДНІ переваги і ОДНИЙ недолік (вистачить для базових питань).
4. Знати "Відносини з іншими патернами" для топ-5 пар: Adapter/Bridge, Composite/Decorator, Strategy/State, Proxy/Decorator, Observer/Mediator.

**Під час захисту:**
- Якщо запитують "застосування Х" — згадай secret weapon: можеш сказати "це використано у моїй ЛР, завдання Y" + перейти до коду.
- Якщо запитують "переваги/недоліки" — є ВЖЕ заготовлені відповіді з лекції.
- Якщо запитують "чим відрізняється від Y" — секція "Відносини з іншими патернами" відповідає прямо.
