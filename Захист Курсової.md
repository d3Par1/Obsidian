## 2. Стек технологій — і чому саме він

```
┌─────────────────────────────────────────────────────────┐
│ Browser (клієнт)                                        │
│   Bootstrap 5  → адаптивна сітка, готові компоненти     │
│   Chart.js 4   → інтерактивні pie/bar/line графіки      │
│   Vanilla JS   → форми, AJAX, валідація                 │
├─────────────────────────────────────────────────────────┤
│ Express 4 (сервер) — MVC                                │
│   Routes       → 9 файлів (accounts, auth, charts, ...) │
│   Models       → 6 файлів (User, Account, Transaction,..│
│   Views        → EJS-шаблони з layout/partials          │
│   Middleware   → helmet, sessions, validation, auth     │
├─────────────────────────────────────────────────────────┤
│ SQLite (БД) через better-sqlite3                        │
│   6 таблиць + 7 індексів                                │
│   Атомарні транзакції через db.transaction()            │
├─────────────────────────────────────────────────────────┤
│ External Services                                       │
│   ExchangeRate API  → курси валют                       │
│   Gmail SMTP        → email-сповіщення                  │
│   Google OAuth      → реєстрація через Google           │
│   Telegram Bot API  → нотифікації + OAuth-логін         │
└─────────────────────────────────────────────────────────┘
```

### Чому **Node.js + Express**?
- Однорідний JS на клієнті й сервері → менше context-switching
- Найпопулярніша зв'язка для Web-програмування в індустрії
- Express — мінімалістичний, дає повний контроль над middleware-pipeline
- **NPM-екосистема** — для кожної задачі (валідація, BCrypt, sessions) є production-ready бібліотека

### Чому **SQLite (better-sqlite3)**, а не PostgreSQL/MySQL?
- Для курсового проєкту з одним користувачем за раз — SQLite ідеальний: zero-config, файл-як-БД
- `better-sqlite3` дає **синхронний** API — простіший код без callback hell або async/await wrap
- WAL-режим (`finance.db-wal`) — паралельні читання при одночасних записах
- Легко продемонструвати на захисті — БД у git'і, не треба піднімати docker

### Чому **EJS**, а не React/Vue?
- Тема — про **повний backend**, а не SPA. EJS — server-side rendering з мінімумом JS
- Простіше демонструвати MVC: маршрут → модель → шаблон → HTML
- Час побудови сторінки <50ms, без bundle-розмірів і hydration

### Чому **Bootstrap 5**?
- Готова адаптивність (`row`, `col-md-6` тощо) — економія часу на CSS
- Темна тема (`data-bs-theme="dark"`) з коробки
- Семантичні класи (`.btn-primary`, `.alert-warning`) — узгоджений UI

## 3. Архітектура додатку

### MVC-патерн
```
        HTTP-запит
            ↓
    [Middleware ланцюжок]
       │ helmet     ← security headers
       │ sessions   ← express-session + connect-sqlite3
       │ auth       ← requireAuth()
       │ validation ← express-validator
            ↓
       [Router]
       │ routes/transactions.js → POST /transactions
            ↓
       [Controller (route handler)]
       │ Викликає Transaction.create(data)
            ↓
       [Model]
       │ models/Transaction.js → db.transaction(() => {...})
            ↓
       [Database]
       │ INSERT INTO transactions ...
            ↓
       res.redirect()  ← PRG pattern (Post-Redirect-Get)
```

### Структура каталогів
```
finance-app/
├─ server.js              — entry point, Express config
├─ db/
│  └─ schema.sql          — 6 таблиць + 7 індексів
├─ models/                — 6 моделей (User, Account, Transaction, …)
├─ routes/                — 9 роутерів
├─ views/                 — EJS-шаблони + layout + partials
├─ public/                — статика (CSS, JS, images)
├─ services/              — currency, oauth, telegram, email
├─ middleware/            — auth, validation, errors
├─ helpers/               — utility-функції
└─ tests/                 — 4 файли смок-тестів (28 кейсів)
```

### Чому MVC, а не layered/clean architecture?
- Express — натуральний MVC: routers = controllers, ejs = views, files in `/models` = models
- Для **навчального проєкту** надмірне розшарування лише ускладнило б код
- 11 пунктів силабусу зосереджені саме на MVC-патернах (роутинг, шаблони, моделі)

## 5.1 Зона Назара — детальний розбір з файлами та патернами

### A) Auth-система — `routes/auth.js` (358 рядків)

Це **найбільший і найскладніший файл моєї зони**. Реалізує 7 endpoint'ів:
| HTTP | Route | Що робить |
|---|---|---|
| `GET` | `/auth/login` | Сторінка логіну |
| `GET` | `/auth/register` | Сторінка реєстрації |
| `POST` | `/auth/register` | Створення нового user'а з валідацією + bcrypt |
| `POST` | `/auth/login` | Перевірка пароля + видача сесії |
| `POST` | `/auth/logout` | Знищення сесії |
| `GET` | `/auth/profile` | Сторінка профілю + timeline подій |
| `POST` | `/auth/change-password` | Зміна пароля (для local-юзерів) |

**Чотири реальні захисти проти атак:**

1. **CWE-384 Session Fixation** — `req.session.regenerate()` після успішного логіну (рядок 103). Це знищує старий session ID і видає новий. Якщо зловмисник підсунув свій cookie до логіну (через XSS) — після `regenerate()` цей cookie стає невалідним.

2. **CWE-208 Timing-attack на існування user'а** — у duplicate-email branch (рядок 144) ми викликаємо `bcrypt.hashSync(password, 10)` **навіть якщо user існує** — щоб час відповіді був однаковий для "юзер є" і "юзер створений". Без цього зловмисник міг би вгадувати email'и за різницею в мілісекундах.

3. **Brute-force через rate-limit** — `express-rate-limit`:
   - Login: **5 спроб/хвилину** з IP, `skipSuccessfulRequests: true` (вдалі не рахуються)
   - Register: 3 спроби/годину
   - Password change: 5 спроб/15 хв

4. **Audit log** — `auth_events` таблиця, кожен event типу `login`, `login_failed`, `register`, `logout`, `password_changed` фіксує `user_id`, `email`, `ip`, `user_agent`, timestamp. На сторінці профілю ці події рендеряться як timeline з UA-parser'ом для розпізнавання браузера/OS.

**Структура completeLogin() — фундаментальний хелпер:**
```js
function completeLogin(req, res, user, { redirectTo, flashMsg, event, provider }) {
  req.session.regenerate((err) => {           // ← новий session ID
    req.session.userId = user.id;             // ← ідентифікація
    req.session.userName = user.name;
    req.session.userEmail = user.email;
    logAuthEvent(req, { event, userId, email, provider });  // ← audit
    res.redirect(redirectTo);
  });
}
```
Викликається з 4 місць: POST /register, POST /login, /google/callback, /telegram/callback — **єдине джерело істини для "юзер залогінений"**.

### B) OAuth-інтеграції

#### B.1 Google OAuth — `services/oauth.js` (70 рядків) + endpoints у `routes/auth.js:210-240`

- Бібліотека: `passport-google-oauth20`
- **Conditional registration** — якщо `GOOGLE_CLIENT_ID` чи `GOOGLE_CLIENT_SECRET` відсутні (`googleEnabled = false`), Google-кнопка ховається у view, callback повертає 503. Це дозволяє розгортати на Render навіть без OAuth-секретів — додаток не падає.
- **Stateless callback** — `{ session: false }` бо я не використовую passport.session(); сесія створюється через мій `completeLogin()` після `passport.authenticate()` вже зверифікувало Google.
- **Account linking** — якщо юзер уже зареєстрований через email/пароль і потім логіниться через Google з тим самим email, `User.linkProvider(existing.id, 'google', googleId, avatarUrl)` приєднує Google-ID до існуючого акаунта (не створює дубль).

#### B.2 Telegram OAuth — `services/telegram.js` (70 рядків)

**Унікальна частина проєкту** — більшість студентських проєктів не реалізують Telegram OAuth, тільки Google.

Telegram Login Widget відправляє юзерські дані з `hash` — HMAC-SHA256 від решти полів, key = `SHA-256(BOT_TOKEN)`. Я перевіряю:

1. **Signature verification** (рядок 32-44):
   ```js
   const secret = crypto.createHash('sha256').update(BOT_TOKEN).digest();
   const computed = crypto.createHmac('sha256', secret).update(dataCheckString).digest('hex');
   crypto.timingSafeEqual(Buffer.from(hash, 'hex'), Buffer.from(computed, 'hex'));
   ```
   - `timingSafeEqual` — **constant-time compare**, протидія timing-attack на знаходження валідного хешу
   - Будь-яке байт-розбіжність → 403, без логіну

2. **Anti-replay через `auth_date`** (рядок 50): payload відхиляється, якщо `auth_date` старіший за 5 хв (Telegram рекомендує ≤24h, я скоротив до 5 хв для більшої безпеки). Також відхиляю timestamps з майбутнього (>30s) — захист від clock skew експлойтів.

3. **Reasons logging** — функція повертає `{ valid: false, reason: '...' }`, де reason ∈ {`missing_hash`, `bad_hash_format`, `signature_mismatch`, `stale_or_future_payload`, `telegram_disabled`}. У `routes/auth.js` ці причини логуються до stderr (commit `f9d3854`) — допомагає діагностувати продакшен-issues.

**Чому Telegram особливо ціна:** Telegram не повертає email (не всі юзери мають публічний email). Я генерую placeholder `${telegramId}@telegram.local` як унікальний ідентифікатор. Цей дизайн дозволяє приєднати Telegram до існуючого акаунта через окрему link-функцію в профілі.

### C) Accounts — `models/Account.js` (89 рядків) + `routes/accounts.js` (128 рядків)

Account = "гаманець" (Картка ПУМБ, Cash UAH, Заощадження USD). 4 CRUD-операції + одна агрегатна.

**Що особливого:**

1. **`getTotalBalance(userId, targetCurrency = 'UAH')` — async з конвертацією валют** (commit `88133da`):
   ```js
   const { rates } = await fetchRates(target);
   return accounts.reduce((sum, a) => {
     const cur = a.currency.toUpperCase();
     if (cur === target) return sum + a.balance;
     return sum + a.balance / rates[cur];  // ← divide, не multiply
   }, 0);
   ```
   - **Чому divide, не multiply:** `fetchRates('UAH')` повертає `{USD: 0.024}` — це означає "1 UAH = 0.024 USD". Щоб конвертувати 500 USD → UAH, потрібно `500 / 0.024 ≈ 20833 UAH`.
   - **Fallback** при відмові API: повертає лише акаунти у target-валюті — graceful degradation замість 500-помилки.

2. **DELETE з referential integrity** (рядки 47-63):
   ```js
   static delete(id, user_id) {
     const tx = db.transaction(() => {
       const txCount = db.prepare('SELECT COUNT(*) ... WHERE account_id = ?').get(id).cnt;
       if (txCount > 0) return { deleted: false, reason: 'has_transactions', count: txCount };
       db.prepare('DELETE FROM accounts ...').run(id, user_id);
       return { deleted: true };
     });
     return tx();
   }
   ```
   - Атомарна перевірка через `db.transaction()` — не можна видалити account, якщо до нього прив'язані транзакції
   - Це **RESTRICT-семантика** на рівні моделі, доповнює FK CASCADE/RESTRICT у schema.sql
   - Route повертає flash-error при `has_transactions` — UX-friendly

3. **Multi-currency seed** — `Account.create()` приймає `currency`, дефолт `'UAH'`, валідація ISO-кода `[A-Z]{3}` у `routes/accounts.js:18`.

### E) Charts API — `routes/charts.js` (49 рядків) + `models/Statistic.js` (68 рядків)

JSON-endpoints для Chart.js. Дві:

#### E.1 `GET /api/charts/categories?month=YYYY-MM`
- Pie-діаграма: витрати по категоріях за вказаний місяць
- Дефолт — поточний місяць. Валідація regex'ом `/^\d{4}-\d{2}$/` — якщо невалід, fallback на current month (тест перевіряє)
- Повертає `{ month, labels: [...], data: [...], colors: [...] }`

#### E.2 `GET /api/charts/monthly?months=6`
- Bar-діаграма: income vs expense за останні N місяців
- N clamped до `[1..12]` — параметр `months=99` обрізається до 6 (тест перевіряє це теж)
- SQL використовує **`strftime('%Y-%m', date)`** для групування по місяцях + **CASE WHEN** для розщеплення income/expense в один прохід:
  ```sql
  SELECT strftime('%Y-%m', date) AS month,
         SUM(CASE WHEN type = 'income' THEN amount ELSE 0 END) AS income,
         SUM(CASE WHEN type = 'expense' THEN amount ELSE 0 END) AS expense
  FROM transactions
  WHERE user_id = ? AND date >= date('now', ? || ' months')
  GROUP BY month ORDER BY month ASC
  ```
- **Один запит замість двох** — навіть якщо в БД 10 000 транзакцій, повертається ~6 рядків за один pass

**Чому це API, а не SSR:** дашборд може перерендерити графік (зміна періоду, фільтри) без перезавантаження сторінки. Chart.js завантажує дані через `fetch('/api/charts/...')` і викликає `chart.update()`.

### F) Currency API integration — `services/currencyService.js` (55 рядків)

- API: `open.er-api.com/v6/latest/{base}` — **безкоштовний, без ключа** (на відміну від ExchangeRate-API.com Pro)
- **In-memory cache** (Map) з TTL = 1 година → одна реальна fetch'а на годину для всього додатку
- `convert(amount, from, to)` — async, повертає число
- `getPopularRates(base)` — для бічного блоку дашборду "Exchange rates" з USD/EUR/PLN/GBP

**Чому in-memory cache, а не БД:**
- Прості об'єкти, малий розмір, не треба persistence (на рестарті переякшеться)
- Уникнули зайвої колонки в SQL-схемі
- Trade-off: при scale до multi-instance треба було б Redis. Для курсової — overkill.

## 6. Ключові технічні рішення

### 6.1 Атомарні транзакції (`db.transaction(fn)`)
Коли користувач створює грошову транзакцію — потрібно одночасно:
1. INSERT у `transactions`
2. UPDATE балансу в `accounts` (відняти витрату або додати дохід)

Це **МАЄ** бути атомарно — інакше при падінні після кроку 1 баланс і транзакції роз'їдуться.

```js
// models/Transaction.js
static create(data) {
  return db.transaction(() => {
    const stmt = db.prepare('INSERT INTO transactions ...');
    const result = stmt.run(...);
    db.prepare('UPDATE accounts SET balance = balance + ? WHERE id = ?')
      .run(amount, accountId);
    return result.lastInsertRowid;
  })();
}
```

**Чому better-sqlite3 ідеальний:** його `.transaction()` обгортка автоматично BEGIN/COMMIT/ROLLBACK залежно від throw.

### 6.2 PRG (Post-Redirect-Get) pattern
Після POST /transactions → `res.redirect('/transactions')` замість `res.render()`.

**Чому:** якщо користувач F5 → не дублюється POST, бо браузер пам'ятає GET-redirect-URL. Класична захист від випадкового подвійного INSERT.

### 6.3 Сесійний контекст через `connect-sqlite3`
- Sessions зберігаються в SQLite-таблиці `sessions` (а не в RAM)
- При рестарті сервера — користувачі залишаються залогінені
- Cookie має `httpOnly: true, secure: true, sameSite: 'strict'`

### 6.4 Adapter-pattern для OAuth-провайдерів
- `services/oauth.js` — спільний інтерфейс
- Google і Telegram — два конкретні провайдери
- Додавання Facebook потребує тільки нового адаптера (Open-Closed Principle)

## 7. Безпека веб-додатку

| Загроза | Захист | Файл |
|---|---|---|
| **SQL Injection** | Prepared statements (`db.prepare('... WHERE id = ?')`) | усі моделі |
| **XSS** | EJS auto-escape (`<%= %>` екранує, `<%- %>` не екранує) | views |
| **CSRF** | `sameSite: 'strict'` на session cookie | `server.js` |
| **Clickjacking, MIME sniff** | `helmet()` middleware | `server.js` |
| **Перехоплення паролів** | bcrypt (cost factor 12) | `models/User.js` |
| **Brute-force** | Логування у `auth_events`, rate-limit | `routes/auth.js` |
| **Session hijack** | HttpOnly cookies, regenerate ID after login | `routes/auth.js` |
| **Невалідні дані** | `express-validator` ланцюжки | `middleware/validation.js` |

### bcrypt — детальніше
- `bcrypt.hash(password, 12)` — saltRounds=12 означає 2^12 = 4096 ітерацій
- Сіль ВКЛЮЧЕНА в результуючий хеш (`$2b$12$<salt><hash>`) — не треба окремої колонки
- `bcrypt.compare()` для перевірки — порівняння в constant time проти timing attacks

## 10. Часті питання комісії (FAQ)

### Q1: Чому не використали ORM (Sequelize, Prisma)?
**A:** Для проєкту з 6 таблицями ORM додає бойлерплейт без виграшу. Prepared statements у `better-sqlite3` дають той самий захист від SQL-injection, що й ORM. Без ORM **легше пояснювати на захисті**, що відбувається — SQL прозорий.

### Q2: Як забезпечена цілісність даних при паралельних операціях?
**A:** SQLite в WAL-режимі підтримує **одночасні читачі + один писач**. Транзакції обертають у `db.transaction(fn)` — це автоматичний BEGIN IMMEDIATE / COMMIT / ROLLBACK. Foreign keys увімкнено через `PRAGMA foreign_keys = ON`.

### Q3: Що буде, якщо ExchangeRate API недоступний?
**A:** `services/currencyService.js` має fallback на кешовані курси (зберігаються в `currency_cache` таблиці з TTL). При мережевій помилці — повертаємо останній відомий курс і логуємо у консоль. Користувач бачить warning-banner "Курси можуть бути неактуальними".

### Q4: Чому SQLite, а не PostgreSQL?
**A:** Курсовий проєкт — single-tenant, з відносно невеликим навантаженням (десятки тисяч транзакцій на користувача). SQLite в WAL-режимі тримає це легко. Для production з 1000+ конкурентних користувачів — мігрував би на PostgreSQL, моделі написано так, щоб SQL-діалект змінити мінімально.

### Q5: Як працює інтеграція з Telegram OAuth?
**A:** Telegram Login Widget дає JS-callback з `id`, `first_name`, `username`, `hash`. Hash перевіряється через HMAC-SHA256 з secret-токеном бота — це доказ, що дані прийшли від Telegram, а не підроблені. Перевірений id зберігаємо в `users.telegram_id`.

### Q6: Чи може користувач бачити чужі дані?
**A:** Усі моделі мають `WHERE user_id = ?` у кожному запиті. Контролер бере `user_id` з `req.session.userId`, який задається тільки при логіні. **Прямого доступу до моделі обходячи сесію немає.** Smoke-test `smoke-auth.js` перевіряє, що неавторизований запит на `/transactions` повертає 302 → `/login`.

### Q7: Як ви розподілили роботу і як перевірити свій внесок?
**A:** Через git: `git log --author="Stepanenko"` показує мої коміти. Кожна фіча — окрема pull request (через мій GitHub-нік). Артем мерджив у main після code review від мене, і навпаки. **Я підготував файл `WORK_SPLIT.md` зі зведенням, хто що писав.**

### Q8: Чому Bootstrap а не власна CSS-система?
**A:** Стилізація — лише 1 пункт з 11 силабусу. Витрачати тиждень на власну сітку коли є production-ready Bootstrap — нераціонально. Унікальність UI — за рахунок темної теми + custom-палітри у `public/css/custom.css`.

### Q9: Що з безпекою сесій?
**A:** Cookie з `httpOnly: true` (JS не читає), `secure: true` (тільки HTTPS на production), `sameSite: 'strict'` (захист від CSRF). Сесії в SQLite, а не в RAM — пережили рестарт. При логіні викликається `req.session.regenerate()` — захист від session fixation.

### Q10: Як ви тестували інтеграцію з OAuth?
**A:** OAuth-flow важко мокати — це 3-stage redirect через зовнішній сервіс. У smoke-тестах перевіряю **callback handler**: GET `/auth/google/callback?code=fake` → ловлю помилку обміну, перевіряю, що сесія НЕ створена. Реальний flow перевірено manually на production-deploy.

### Q11: Що буде, якщо база пошкодиться?
**A:** SQLite в WAL-режимі автоматично відновлюється з `.wal`-файла. У production додаткові заходи: щоденний `sqlite3 finance.db .backup backup.db` через cron. На фінансовому продакшені користувався б ще `pg_dump`-rotation.

### Q12: Як ви валідуєте суму транзакції?
**A:** `express-validator`: `body('amount').isFloat({ min: 0.01, max: 1_000_000 })`. Без знаку — знак визначається типом транзакції (`income`/`expense`). На клієнті — `<input type="number" step="0.01" min="0">`. На SQL — `CHECK(amount > 0)`.

---

### Q13: Чому ви викликаєте bcrypt навіть для неіснуючого юзера при duplicate-email?
**A:** Це **захист від timing-side-channel** (CWE-208). Якщо повертати 422 одразу, коли email уже існує — час відповіді ~5 мс. Якщо створювати нового — bcrypt займає ~80 мс. Зловмисник міг би вгадувати, які email'и зареєстровані, за різницею в часі відповіді (`Was email in DB? Yes if response > 50 ms`). Тому в branch'і "duplicate" я викликаю `bcrypt.hashSync(password, 10)` навіть якщо result буде відкинутий — щоб час відповіді був завжди ≥80 ms незалежно від наявності user'а. **Файл:** `routes/auth.js:142-150`.

### Q14: Як verification роботить для Telegram OAuth?
**A:** Telegram Login Widget відправляє юзерські поля (`id`, `first_name`, `auth_date`, тощо) + `hash`. Hash — це **HMAC-SHA256** від решти полів (відсортованих + join'ом через `\n`), а ключ — **SHA-256(bot_token)**. Я recompute hash на сервері й порівнюю через `crypto.timingSafeEqual` — constant-time, щоб запобігти timing-attack. Додатково: відхиляю payload з `auth_date` старішим за 5 хв (anti-replay). **Файл:** `services/telegram.js:32-52`. Будь-яке байт-розходження hash → 403 + лог події `login_failed_telegram`.

### Q15: Чому Google OAuth не використовує passport.session()?
**A:** Бо я зробив **свій post-auth handler** `completeLogin()`, який робить `req.session.regenerate()` — це захист від session fixation (CWE-384). Якщо б я використовував `passport.session()`, він би серіалізував user у вже існуючу сесію, не регенеруючи її. Тоді зловмисник, який підкинув юзеру свій session cookie (через XSS до логіну) — отримав би валідну сесію після того, як юзер логінився Google'ом. Моє рішення: `passport.authenticate('google', { session: false })` + після успішного callback'а — `completeLogin()` з регенерацією. **Файл:** `services/oauth.js:54-57` + `routes/auth.js:100-115`.

### Q16: Як працює in-memory cache в currencyService?
**A:** `services/currencyService.js` тримає `Map` з ключем = базова валюта (`'UAH'`, `'USD'`, …) і значенням = `{ data, fetchedAt }`. На кожен виклик `fetchRates(base)` перевіряю `Date.now() - fetchedAt < CACHE_TTL_MS (1 година)`. Якщо так — повертаю кеш, **жодного HTTP-запиту**. Якщо ні — fetch з API, оновлюю кеш. **Trade-off:** при рестарті сервера кеш втрачається (`new Map()`); для one-instance Render free це OK. При multi-instance scale потрібен Redis-cache. **Файл:** `services/currencyService.js:8-33`.

### Q17: Як ви робите конвертацію валют у total balance?
**A:** Спрощено: для кожного акаунта `(balance, currency)`, якщо `currency` збігається з target — додаю напряму, інакше — `balance / rates[currency]`. **Чому ділення:** `fetchRates('UAH')` повертає `{USD: 0.024}` — це "1 UAH = 0.024 USD". Щоб конвертувати 500 USD → UAH, треба `500 / 0.024 ≈ 20833 UAH`. Один fetch на годину, потім чиста математика. **Файл:** `models/Account.js:25-50`. Якщо API недоступний — fallback: повертаю суму лише акаунтів у target-валюті (graceful degradation, не 500-помилка).

### Q18: Чому ви розділили Charts на окремий API роутер, а не рендерите HTML?
**A:** Дашборд має 2 графіки (pie за категоріями + bar місячні income/expense). Якби я рендерив їх SSR — кожна зміна фільтра (наприклад, інший місяць) потребувала б повного перезавантаження сторінки. Розділивши на JSON-endpoint'и (`/api/charts/categories`, `/api/charts/monthly`) — Chart.js на клієнті робить `fetch()` + `chart.update()`, **без перезавантаження**. Це сучасний AJAX-патерн, не SPA, але дає кращий UX. **Файли:** `routes/charts.js`, `public/js/dashboard-charts.js`.

### Q19: Чому SQL-запит у Statistic.getMonthlyTotals() використовує CASE WHEN?
**A:** Класична трюк "**pivoting**": замість двох окремих запитів (один для income, один для expense), один запит з `SUM(CASE WHEN type='income' THEN amount ELSE 0 END)` повертає обидва значення в одному рядку. Це означає **2× менше I/O** до БД. SQLite добре оптимізує цей патерн з індексом `(user_id, date)`. **Файл:** `models/Statistic.js:21-31`.

### Q20: Як працює endpoint /admin/reseed і чому він потрібен?
**A:** Render free tier стирає файлову систему при кожному redeploy. Без захисту — `finance.db` порожня, не можна залогінитись. Два механізми: (1) `AUTO_SEED_DEMO=true` env var → boot-time auto-seed якщо `users` таблиця порожня; (2) `GET /admin/reseed?token=$ADMIN_TOKEN` — token-protected ручний reseed для defense day, якщо щось зіпсувалось. Токен генерується Render'ом автоматично (`generateValue: true` у `render.yaml`). Без токена endpoint повертає 403. **Файли:** `routes/admin.js`, `scripts/seed-demo.js`, `render.yaml:18-22`.

```sql
-- Останні 10 транзакцій з context
SELECT t.id, t.amount, a.currency, a.name AS account, c.name AS category, t.created_at
FROM transactions t
JOIN accounts a ON t.account_id = a.id
JOIN categories c ON t.category_id = c.id
WHERE t.user_id = (SELECT id FROM users WHERE email = 'demo@kpi.ua')
ORDER BY t.created_at DESC LIMIT 10;

-- Сума балансів по валютах
SELECT currency, SUM(balance) AS total
FROM accounts
WHERE user_id = (SELECT id FROM users WHERE email = 'demo@kpi.ua')
GROUP BY currency;

-- Перевірка foreign keys (показує цілісність)
PRAGMA foreign_key_check;
```

### 12.4 Render deploy — auto-seed і admin endpoint

Render free tier стирає файлову систему при кожному redeploy. Без захисту — `finance.db` порожня, увійти неможливо. Два механізми:

**1. AUTO_SEED_DEMO (boot-time, автоматичний)**

У `render.yaml` встановлено env var `AUTO_SEED_DEMO=true`. На старті сервер перевіряє чи таблиця `users` порожня — якщо так, виконує `seedDemo()` автоматично. Не перезаписує реальні дані: щойно хтось зареєструється, auto-seed більше не спрацьовує.

**2. Admin endpoint (manual reseed з URL)**

URL: `https://<твій-render-домен>/admin/reseed?token=<ADMIN_TOKEN>`

Якщо під час захисту хтось зіпсує демо-дані — заходиш на цей URL з браузера → видає HTML-сторінку з підтвердженням → робиш login заново. **Токен генерується Render'ом** автоматично при першому деплої:
1. Зайди в Render Dashboard → твій сервіс → Environment
2. Знайди `ADMIN_TOKEN` → скопіюй значення
3. Збережи в Obsidian або в нотатці на телефоні — не в самому репо!

**Sister endpoint для перевірки стану:**
```
GET /admin/status?token=<ADMIN_TOKEN>
→ {"ok":true,"counts":{"users":1,"accounts":3,"transactions":15,"categories":15},"demoUserExists":true}
```

Можна використати для quick-check, що БД у потрібному стані перед demo.

### 12.5 Колаборація з Артемом

Якщо Артем фіксить щось і ти хочеш бачити його зміни — він пише в БД через UI, ти у DB Browser натискаєш ▶ Execute SQL → бачиш свіжий рядок. **WAL-режим SQLite дозволяє concurrent читачів + один писач**, тому твоя read-only сесія не блокує його запис.

Це **обов'язково згадати на захисті** — якщо комісія спитає "як ви тестували cooперативно?":
> "Через DB Browser у read-only режимі — SQLite в WAL-режимі підтримує конкурентне читання, тому коли Артем додавав транзакції з браузера, я в IDE одразу бачив новий запис у `transactions`-таблиці."
